# Evidence Artifact Retention Policy

## Purpose
This policy prevents accidental commits of generated artifacts and sensitive raw evidence while preserving intentional, auditable tracking of sanitized decision-grade reports.

## Scope
Applies to local artifacts generated by:
- Playwright and browser audits
- OCR extraction flows
- TPS packet/PDF generation checks
- CI/production verification evidence bundles

## Classification Rules
1. Generated test artifact:
- Example: `apps/web/test-results/`, `playwright-report/`
- Rule: local-only, ignored by git.

2. Raw evidence artifact (sensitive-by-default):
- Example: screenshots, packet ZIPs, PDFs, HAR/trace/network logs, OCR raw outputs under `docs/reports/evidence/`
- Rule: local-only by default, ignored by git.

3. Durable evidence summary (trackable):
- Example: `FINAL_*_REPORT.md`, curated `README.md` in an evidence run folder
- Rule: may be tracked only after sanitization review and operator approval.

4. Local benchmark/debug report:
- Example: `reports/booklet-stability-*/`
- Rule: local-only by default, ignored by git.

5. CSV reports:
- Example: `reports/booklet-synthetic-multisample-*.csv`
- Rule: not ignored globally; track only if sanitized and decision-grade.

## Sensitive-By-Default Categories
Treat the following as sensitive unless explicitly reviewed and sanitized:
- OCR payloads and extraction traces
- Generated immigration packet ZIP/PDF artifacts
- Screenshots of user flows that may include personal data
- Network captures (`.har`, `.trace`, request/response logs)
- Any artifact that can reveal user/customer identity data

## PII/OCR Handling Rules
- Do not track raw OCR text dumps containing personal data.
- Do not track raw packet bundles (`.zip`) or generated form PDFs by default.
- Do not track screenshots that include user document content or account identifiers.
- If a report references sensitive source artifacts, keep references high-level and avoid embedding raw values.

## Promotion Workflow: Local Artifact -> Tracked Sanitized Evidence
1. Confirm business need:
- The artifact must support an engineering decision, incident analysis, or release gate.

2. Sanitize:
- Remove/obfuscate personal identifiers and raw document extracts.
- Replace raw payload dumps with summaries and field-level conclusions.

3. Reduce:
- Keep only decision-grade facts (run id, command, outcome, key metrics, blocker, next step).

4. Publish approved summary:
- Prefer `docs/reports/evidence/<run>/FINAL_*_REPORT.md` or a concise `README.md`.
- Keep raw binaries/logs local or archive outside git per owner policy.

5. Verify before commit:
- Confirm no raw sensitive files are staged.
- Confirm tracked report contains evidence links/paths without exposing secret values.

## Operational Guardrails
- No destructive cleanup in automated sessions without explicit approval.
- No secret values in tracked reports.
- No DONE claims without evidence-backed verification.
